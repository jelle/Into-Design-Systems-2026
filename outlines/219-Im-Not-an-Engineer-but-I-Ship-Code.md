# I'm Not an Engineer but I Ship Code
**Speaker**: Freya Sockman, Full Stack Product Designer, Relevance AI
**Conference**: Into Design Systems AI Conference 2026

## Key Ideas

1. **2026 is the tipping point for designer-engineers** — The double diamond is expanding: designers are now owning technical scoping, writing tickets, and shipping full front end implementations, not just handing over specs.

2. **Everything on the internet is just text files** — This reframe is the biggest unlock: you don't need to know how to code, but once you realize all code is text, you can communicate with AI tools to manipulate it fluently.

3. **The 70/20/10 rule: plan first, implement later** — Spend 70% of your time documenting and understanding the codebase, 20% on implementation, 10% on polish. Deep planning prevents bugs and failed builds.

4. **Turn your Figma flows into MD files** — Export frames as images, add voice-to-text narration explaining the flow, then ask the AI to write it all into a markdown file. This gives AI rich, persistent context it can reference at any time.

5. **Think like a solution architect, not a pixel pusher** — Before touching code, map out current components, risks, downstream impacts, and backend dependencies. AI can help surface these if you ask conversationally.

6. **Cursor is the command center for UI precision** — It lets you select elements directly in chat, view on localhost, and edit properties in real time. Preferred tool for work; Claude Code for personal projects and multi-agent parallelism.

7. **APIs and MCPs are just bridges — not scary** — Think of them as post offices: every platform has an API key, and AI tools can talk to Notion, Linear, or anything else from your command center without you ever switching apps.

8. **GitHub is Dropbox for code** — Branch = draft, commit = save, push = upload, pull request = peer review. You don't need to memorize git commands; just tell cursor "commit my changes" and it handles it.

9. **Slot into the existing engineering review workflow** — Designers shipping code should go through the same PR review process as engineers. When you get technical comments, paste them into cursor and let it fix them. No deep understanding required.

10. **Figma is still alive — just not the whole story** — Figma remains valuable for high-level flows, diagrams, and stakeholder communication. But for complex interactions and edge case exploration, prototyping in cursor is faster and more accurate.

11. **Ship the MVP, not the optimal state** — Map out the two-function minimum viable version vs. the full ideal flow. Ship the MVP first. Iterate toward optimal. Prevents scope creep and over-building.

12. **Vibe coding is the gateway to a much bigger toolkit** — Beyond shipping features, these skills unlock automations, second-brain tools, API integrations, and AI voice assistants. The billboard onboarding flow was just the beginning.

13. **Know when to hand over vs. when to ship** — If there is significant backend work, hand it to an engineer. If it is purely frontend with no new backend build, a designer can and should own it. Discernment matters more than ambition.

14. **AI coding tools make designers more hireable** — Job ads in 2026 are explicitly looking for vibe coding flexibility. These skills are not a bonus anymore; they are becoming a baseline expectation.
