# Encoding Governance on Agentic Design Systems
**Speaker**: Christian (Chris), Design Engineer, NARA
**Conference**: Into Design Systems AI Conference 2026

## Key Ideas

1. **Governance used to be a luxury — now it is an output** — Design system teams are chronically understaffed (60% have zero pipeline automation, 61% feel understaffed). The old model of enforcing governance manually cannot scale. With agentic infrastructure, governance becomes a byproduct of how the system is built.

2. **The core value of a design system is cognitive offloading** — When the system works, it removes solved problems from every consumer's mental load. Spacing, color, typography, component behavior — those decisions should not require attention twice.

3. **The "one source of truth" should be the one that cannot die** — Different roles have different truths: Figma for designers, Storybook for devs, Excel for PMs. The real source of truth should live in code contracts — explicit, enforceable, not dependent on a platform staying alive.

4. **Four structural layers: tokenization, intent, index, strategy** — Each layer eliminates a different kind of AI failure: tokens eliminate infinite decision space, metadata eliminates interpretation, the index eliminates exploration, and strategy (agent instructions + rules + skills) eliminates improvisation.

5. **The token auditor goes beyond linting** — It does not just check "are we using a token?" but "are we using the correct token for this context?" Elevation hierarchy, semantic color models, and typography rules are encoded as explicit constraints, producing comparable, schematized reports.

6. **Component metadata is a TypeScript contract, not prose documentation** — Metadata files include use cases, composition patterns, anti-patterns, and accessibility considerations. AI reads the header first, then drills into details — same as how a developer scans an API reference.

7. **The codebase index gives AI a map instead of a maze** — A script generates a world map of the full library: every component, every token category, every relationship. AI consults the map instead of searching. Regenerating it takes 10 seconds after any new component is added.

8. **Skills are reusable, project-portable capabilities** — Skills perform specific operations (scaffold a component, audit tokens, generate documentation) and can be pointed at any project. They encode the methodology so AI follows the same process every time, on any codebase.

9. **The infrastructure cost is upfront, the compound savings are ongoing** — Initial tests showed the agentic system was slower and used more tokens per task. But each task benefits from the existing map, so iteration cost drops dramatically. A design system health report that costs $169/month as a SaaS tool costs $0.20 to run on demand.

10. **Documentation is generated from the codebase, not written separately** — A documentation skill aggregates component logic, metadata, and design specs into a documentation website (built in Astro, generated by Claude). No manual sync required.

11. **Consumer vs. maintainer — two distinct agent roles** — A consumer understands components and implements them. A maintainer audits the codebase, identifies drift and tech debt, proposes improvements, and enforces contracts. Both roles can be AI-assisted once the infrastructure exists.

12. **Solo designer on a team of 11 — infrastructure makes it possible** — Chris is the only designer at NARA. With the agentic layer, the dev team uses the same scaffolding skills to build components outside the design system — and those components automatically conform to encoded standards, making them easy candidates for promotion into the system.

13. **MCPs make sense for larger teams needing remote access; skills for local work** — If you have direct codebase access, skills are more efficient and flexible. MCPs are better when teams need a shared, managed interface to tooling they cannot directly clone.

14. **Figma is becoming a bottleneck, not a foundation** — As the sole designer shipping at this speed, Figma now slows Chris down. It has shifted to a tool for visual diagrams and flow overviews, while the actual component reality lives in code. Prototyping happens through branches, not frames.
